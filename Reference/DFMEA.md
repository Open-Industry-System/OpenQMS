# FMEA介绍及历史
FMEA代表什么？（Failure Mode and Effects Analysis)
       FMEA是一套标准化方法，用以系统化地进行风险分析，并在产品开发阶段避免风险或者将风险最小化。从1970年代开始，FMEA在汽车行业进行运用。

       FMEA是
系统性的定义产品的性能要求和功能要求，以便于发现潜在的失效模式——在产品开发阶段尽可能早的进行
评价功能失效的潜在影响，并且通过制定预防措施来避免失效原因发生或者制定探测措施来检测功能失效的产生。
通过制定合适的预防措施避免质量，保修及售后成本
保证整个供应链完整的追溯性
记录公司的专业知识（例如，通过经验教训库）
是一种常规的方法，用来将过往经验集成在现有设计中

# FMEA介绍及历史
FMEA 的目标
目标：FMEA的目标是确定产品功能或过程步骤以及相关的潜在故障模式、失效后果和原因。除此之外，他被用来评估预防和探测控制是否充分、并建议采取额外行动以降低风险。
FMEA通过分析组件之间的交互和接口（包括功能和故障相关性）来帮助描述产品或过程。它支持功能开发、测试计划和过程控制计划。
FMEA可以帮助组织实现以下目标：
提高质量、可靠性、汽车产品的可制造性、可维修性和安全性
支持系统、子系统和组件之间的逐级关联和一致性
降低保修和商誉成本
就产品责任而言，提供产品和过程风险分析的证据
减少后期变更
发布的产品无缺陷
内外部客户以及供应商之间进行有针对性的沟通
形成内部知识库
零件、系统和车辆对法规符合性
确保组件、系统和车辆之间的层级，连接和接口被识别。

# 范围定义
范围定义为DFMEA制定过程的第一步，即确定项目规划之5T：
inTent-----为什么要做FMEA
Timing----什么时候完成
Team------需要哪些人
Task-------做哪些工作
Tool-------如何进行分析

inTent-----为什么要做FMEA
情形1: 新设计、新技术或新过程
    FMEA 的范围包括完整的设计、技术或过程。
情形2: 现有设计或过程的新应用
    FMEA 的范围包含新环境、新场地、新应用或使用概况（包括工作周期、法规要求等）下的现有设计或过程。FMEA 的范围应关注于新环境、新场地或新应用对现有设计或过程的影响。
情形3: 对现有设计或过程的工程变更
新技术开发、新要求、产品召回和使用现场失效可能会需要变更设计和/或过程。在这种情况下，可能需要对FMEA 进行审查或修订。

# 结构分析
目的：
设计结构分析的目的是将设计识别和分解为系统、子系统、组件和零件，以便进行技术风险分析。
步骤二“结构分析”中定义和信息将被用于步骤三“功能分析”。结构分析中缺少的设计要素（项目），在功能分析中也会缺少。
系统结构：
系统结构由系统要素组成。根据分析的范围，设计结构的系统要素可以由系统、子系统、装配件和组件构成。
复杂的结构可以分为若干子结构（工作包）或不同层次的方块图，并根据组织起因分别进行分析，或确保足够清楚。
系统有一个边界，将自身与其他系统和环境分开。其与环境的关系由输入和输出决定。
在FMEA分析时，需要考虑：最终用户，装配和制造
了解这些顾客有助于更好地定义功能、要求和规范，并有助于确定相关的失效模式的影响。
系统结构的可视化有助于DFMEA团队进行结构分析。团队可以使用各种工具来实现这一点
方块图/边界图
结构树

# 结构分析
方快图/边界图
方块图/边界图 是一种有用的工具，用于描述考虑中的系统及其与相邻系统、环境和顾客的接口。它是一种图表展示法，为结构化的头脑风暴提供指导，并有助于分析系统接口，从而为设计FMEA打下基础。

方块图／边界图的制作可包括以下步骤：
描述组件和特性
调整方块以显示相互间的联系
描述连接
增加接口系统和输入（人员和物件）
确定边界（团队控制范围内有哪些零件？哪些是新的或修改过的？)
    只有团队设计或控制的零件才放在边界内。边界图中的方块比正在分析的层次低一级。可以对边界图中的方块做标记，以显示分析范围之外的零件。
增加相关细节以便确定图表

正确构建的方块图/边界图可为参数图（P图）和FMEA提供详细信息。尽管方块图/边界图的详细程度可以不同，但要标识出主要要素，并了解它们如何相互作用，以及它们如何与外部系统相互作用，这一点很重要。

# 结构分析
接口分析
    接口分析描述系统要素之间的交互作用。接口分析中的信息可为设计 FMEA 提供有价值的输入，例如主要功能或接口功能，这些功能将与相邻系统和环境影响导致的潜在失效起因／机理一起进行分析。

接口有五种类型：
物理连接（如：支架、螺栓紧固、夹紧和其各种连接）
材料交换（如：气压、液压油或任何其它液体或物料的交换）
能量传递（如：热量、摩擦或运动传递，如链条及齿轮）
数据传递 （如：计算机输入或输出、线束、电信号或任何其它类型的信息交换、网络安全项目）
人-机（如：控制、开关、镜子、显示镜、警告、座位、出入口）

接口分析将方块图/边界图中显示的所有内、外部接口发生的关系性质（强/弱/无、有益/有害）和（物理、能量、信息或材料交换）形成文件。
接口分析中的信息可为设计FMEA提供有价值的输入，例如主要功能或接口功能，这些功能将与相邻系统和环境影响导致的潜在失效起因/机理一起进行分析。接口分析还为参数图（P图）提供理想功能和噪音因素的输入。

# 功能分析
目的：

产品或过程功能可视化
将相关要求与顾客（内部和外部）功能关联
将要求或特性与功能关联
工程团队（系统、安全和组件）之间的协作
失效分析步骤的基础
该结构提供了基础，以便能够就每个系统要素的功能和要求进行单独分析。
为此，有必要全面了解系统及其运行条件和环境，如热、冷、灰尘、溅水、盐、冰、振动、电气故障等。

# 功能分析
功能：
功能描述了项目/系统要素的预期用途。
一个功能被分配给一个系统要素，一个结构要素也可以包含多个功能。
功能的描述需要清晰准确
推荐的短语格式为：一个“行为动词”后加一个“名词”，表示可测量的功能。功能应该是现在时态，并使用动词的基本形式（交付、包含、控制、组装、传输）
例如：传输动力，包含液体、控制速度、传递热量、标记黑色
功能描述了一个项目/系统要素的输入和输出之间的关系，目的是完成一个任务。
一个项目除了其主要功能外，可以评估的其他功能包括接口功能、诊断功能和可服务性功能等辅助功能。
明确区分产品的基本功能、详细要求、设计约束条件，例如（刹车系统）：
基本功能：按需刹车
详细要求：车辆在干燥的路面上以80km/h的速度行驶，必须在50m范围内停

# 风险分析
评估：
应评估每一个失效模式、起因和影响，以便对风险进行估计。评估风险的级别标准如下：

从1到10的评估数字分别用于S,O,D，其中10的风险贡献最高。

将一个团队的FMEA和另一个团队的FMEA的评级进行比较时不合适的，及时产品/过程看起来是相同的。因为每个团队的环境都是独一无二的，因此他们各自的评级将是独一无二的（也就是说，评级是主管的）。

探测度（D）：
探测度评级（D）是对探测控制有效性的估计，用于考量在放行生产前探测措施对失效起因或失效模式探测的有效性。
探测评级与最有效的探测控制相关。
探测度是一个相对的评级，且在单个FMEA范围内进行评级，它不考虑严重度或频度。
应使用表中的标准对探测度进行评估。这个表格可以用公司常用的探测方法进行补充。FMEA项目团队应该就评估标准呵评级体系达成一致，即使针对个别产品分析作了修改，该标准和体系也是一致的。
探测度评级最初是对任何尚未被证实的控制的有效性进行的预测。在探测控制完成后，可以对其有效性进行验证和重新评估。

探测失效起因或失效模式的最有效测试是什么？
探测措施在探测失效时需要使用什么配置文件？对探测周期有什么要求？
探测失效需要多少样品？
探测该起因/失效模式的测试程序是否已得到证明？
探测能力：设计验证的试验能力
探测时间：设计开发过程的关键时间节点

# 优化
如果团队决定不需要进一步采取措施，则在备注部分写上“无需进一步的措施”，以表示风险分析已完成。
优化的最有效顺序如下：
修改设计以消除或减少失效影响（FE）
修改设计以降低失效起因（FC）的频度（O）
提高探测（D）失效起因（FC）或失效模式（FM）的能力
在发生设计修改的情况下，所有受影响的设计要素都要重新评估。
在概念变更的情况下，FMEA的所有步骤都要针对受影响的部分进行评审。这是必要的，因为初始分析是基于不同的设计概念

| 改善与降低 | 行动的内容 | 改善的目的 |
| --- | --- | --- |
| 严重度（S） | 改变设计 | 消除或减少故障模式的严重性 |
| 频度（O） | 更改设计或是改进工程规范设计防错 | 防止原因或失效的发生 |
| 探测度（D） | 增加或更改设计验证 设计更改以提高探测可能性 修订测试计划 | 发现失效原因并采取纠正措施 检测失效模式是否发生并纠正 |

# 优化
责任分配
每个措施都应该有责任人和与之相关的目标完成日期。
负责人应确保措施的状态保持更新。如果措施被确认，那么该负责人也要对措施的实施情况负责。
应记录预防和探测措施的实际完成日期，包括措施实施的日期。
目标完成日期应切合实际（例如，按照产品开发计划，在过程验证之前、在生产开始之前）。

# 优化
措施的状态
    措施的状态建议分为以下几类：
尚未确定---没有确定的措施。
    尚未决策（可选）---措施已经确定，但还没有决定。则会能够在创建决策文件。
    尚未执行（可选）---一堆措施做出决定，但尚未执行。
已完成---已完成状态是指措施已经被执行，其有效性以及被证明和记录，并已经进行了最终评估。
不执行---当决定不执行某项措施时，就会分配“不执行”的状态。如果实践和技术限制超出当前能力，就会发生这种情况。
     只有当FMEA团队评估了每个项目的措施优先级，并接受风险水平或记录措施结束       时，FMEA工作才算完成。
如果“不采取措施”，那么“措施优先级”就不会降低，失效风险就会继续进入产品设计。
对于具有开放性目标的措施，需要以书面形式将其关闭。

# 优化
措施有效性的评估
当措施完成时，频度和探测度将重新评估，一个新的措施优先级可能要被确定。新的措施将获得初步措施优先级评估，作为对有效性的预测。
然而，该措施将一直保持“尚未执行”的状态，直到其有效性得到测试为止。测试完成后，初步评估必须得到确认或在必要时调整。然后，措施状态从“尚未执行”改为“已完成”。
重新评估应基于采取的预防和探测措施的有效性，并且新的值应基于设计FMEA频度和探测度评级表中的定义。
持续改进
DFMEA是设计的历史记录。因此，初始严重度、频度和探测度（S、O、D）编号需显示可见，或至少可作为历史记录一部分使用和访问。分析完成后将形成一个存储库，能够记录过程决策和设计改进的进展。然而，对于基础、系列或一般DFMEA，初始的严重度、频度和探测度（S、O、D）评级可能被修改，因为这些信息在特性应用中被用作特定应用分析的起点。

# 结果文件化
目的
结果文件化步骤的目的是，针对FMEA活动的结果进行总结和交流。

将结果文件化的主要目标是：
对结果和分析结论进行沟通
建立文件内容
记录采取的措施，包括对实施措施的效果进行确认、采取措施后进行风险评估
在组织内部，以及与客户和/或供应商之间（如需）针对降低风险的措施进行沟通
记录风险分析和风险降低到的可接受水平。

客户接口FMEA（如果客户与SMTS销售之前达成协议）
