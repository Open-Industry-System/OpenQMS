**更新日期**: 2026-05-25

**参考依据**:

## 1. 测试目标与范围

本测试方案旨在验证 OpenQMS Phase 1 所涵盖的“基础平台与核心模块”的功能完备性、业务逻辑正确性、性能及安全性。

**核心测试范围包括：**
- **基础平台**：RBAC 用户权限控制、产品线(Product Line)全局数据隔离、核心基础防抖与并发控制、底层 AuditLog 审计日志。
- **质量前置策划 (APQP/FMEA)**：DFMEA 编辑器与生成规则引擎 (七步法)、PFMEA、控制计划(CP) 编辑与联动、特殊特性与产品安全特性管理、FMEA/CP 版本与变更比对管理。
- **过程质量控制 (SPC/MSA)**：MSA 计算引擎及管理、SPC 控制图 (v1.0 计量型 + v1.1 计数型 + 控制限多版本 + 八大判异规则)。
- **质量问题闭环处理**：8D / CAPA 工作流及状态流转。
- **体系运行与监控**：质量目标分解、内部审核发现项管理、管理评审自动汇总、可视化仪表盘 KPI。

## 2. 测试策略与分层

### 2.1 自动化测试层级
1. **单元测试 (Unit Testing)**：
   - **前端 (Vitest)**：防抖 Hook、FMEA AP 判定逻辑、SPC 判异规则纯函数、权限指令校验。
   - **后端 (Pytest)**：MSA 算法引擎的数值准确性、数据库增删改查、API RBAC 拦截、数据校验层。
2. **集成测试 (Integration Testing)**：
   - API 与数据库的关联测试：确保外键关联和 JSONB 图结构的完整性。
   - 跨模块流转测试：如 8D 触发表单转接、内审问题自动生成 CAPA。
3. **端到端测试 (E2E Testing)**：
   - 使用 Playwright 模拟用户操作全业务闭环，涵盖高频主要场景。

### 2.2 非功能性与交互体验 (UI/UX) 测试策略
1. **交互体验与视觉 (UI/UX)**：操作反馈的及时性（如 Loading 状态、骨架屏、Toast 提示），响应式布局兼容性，数据复杂表格的交互流畅度，色彩与排版的视觉一致性。
2. **性能与并发测试**：大数据量节点的 FMEA 虚拟渲染 (Virtual Scroll)、多人同时编辑 8D 表单的并发锁/防抖测试。
3. **安全性测试**：RBAC 越权访问测试、JWT 刷新测试、注入拦截、AuditLog 不可篡改性。
4. **隔离性测试**：切换产品线 (DC-DC-100) 时，全局数据的查询可见性必须强隔离。

## 3. 核心业务闭环测试场景 (UAT 场景)

### 场景一：新品研发与质量先期策划 (DFMEA -> PFMEA -> CP 联动)
| 测试步骤 | 操作指南 (操作) | 预期结果 (结果) |
| --- | --- | --- |
| **Step 1: 创建 DFMEA** | 在系统中新建一个 DFMEA，按七步法完成结构树与功能树分析，并标识失效链。在风险分析阶段，将某些失效原因标记为“产品安全特性 (CC/SC)”。 | 结构与功能树渲染正常。AP计算正确。系统准确标记特殊特性图标。 |
| **Step 2: 安全特性审批** | 提交 DFMEA 进行审批，触发针对安全特性的专属会签流程。Admin或质量经理登录并点击通过。 | 会签负责人收到审批通知；审批通过后 DFMEA 状态变更为“已发布”，安全特性记录进入全局特殊特性库。 |
| **Step 3: 关联 PFMEA** | 进入 PFMEA 模块，选择“从 DFMEA 继承”，勾选对应的节点并生成 PFMEA 结构。 | 失效模式、严重度(S) 评分和安全特性标识成功继承至 PFMEA 对应步骤，且无法在 PFMEA 中被随意篡改。 |
| **Step 4: 生成控制计划 (CP)** | 在已发布的 PFMEA 详情页，点击“一键生成控制计划”。 | 成功生成同版本的 CP，产品/过程特性和控制方法 (预防/探测) 字段双向同步，CP 表格正确展示 CC/SC 标识。 |
| **Step 5: 版本对比** | 在 PFMEA 模块点击“新建修订版”，修改某失效原因后保存发布。 | 旧版本被自动锁定(只读)，版本号递增。点击“版本对比”，被修改的节点用绿色/红色高亮显示差异。 |

### 场景二：生产过程质量异常及纠正预防措施流转 (MSA -> SPC -> 8D)
| 测试步骤 | 操作指南 (操作) | 预期结果 (结果) |
| --- | --- | --- |
| **Step 1: MSA 量具分析** | 在 MSA 模块新增一个量具，录入标准 10个零件x3个测量员x3次测量的 GR&R 数据。 | 系统成功生成 GR&R 分析图表。%GR&R < 10% 提示接受；若 > 30% 提示拒收。 |
| **Step 2: SPC 数据判异** | 在 SPC 模块选择一条计数值控制图 (P图)，连续手动录入 7 个落在中心线上方的测试点。 | 图表实时渲染数据点。录入第 7 个点后，系统触发“连续 7 点位于中心线同侧”判异规则，该点标红，界面弹出预警提示。 |
| **Step 3: 触发 8D 流程** | 点击标红的 SPC 异常点，在弹出的菜单中选择“发起 8D 改进”。 | 系统自动创建一个 8D 表单，D2 (问题描述) 自动填入 SPC 异常的上下文(控制图名称、判异规则、发生时间)。 |
| **Step 4: 并发写库** | 打开两个浏览器窗口登录同一账号，同时对 8D 的 D4(根本原因) 进行输入并快速点击外部触发保存。 | 防抖机制生效，只有最后一次合法输入被写库，不产生并发覆盖引发的脏数据或网络报错。 |
| **Step 5: 8D 闭环** | 填写 D5 (预防措施) 并在 D7 中确认更新 PFMEA。点击“关闭 8D”。 | 8D 状态变更为“已关闭”。后台 AuditLog 精确记录 8D 所有字段的修改和状态流转轨迹。 |

### 场景三：体系闭环与管理审核 (质量目标 -> 内审 -> 管理评审)
| 测试步骤 | 操作指南 (操作) | 预期结果 (结果) |
| --- | --- | --- |
| **Step 1: 目标分解** | 管理员创建一个“公司级”一级目标，将其拆分为二级目标分配给不同产品线，点击下发。 | 树状目标结构显示清晰。产品线负责人收到通知，二级目标状态变为“待执行”。 |
| **Step 2: 内审转 CAPA** | 审核员在内审模块记录一个“严重不符合项”，点击操作栏的“转为 CAPA”。 | 成功生成关联的 CAPA 单。内审发现项的状态同步变为“CAPA 处理中”。 |
| **Step 3: 管理评审汇总** | 新建一次年度管理评审，设置统计周期为本年度，点击“自动获取体系数据”。 | 系统正确汇总并图表化展示年度质量目标达成率、内审不符合项分布、以及 CAPA 解决率等数据包。 |
| **Step 4: 追踪效果** | 在管理评审的输出项中添加一条“纠正/预防改善措施”，分配负责人并设置截止日期。 | 该措施生成独立的追踪项，仪表盘上针对负责人的“待办任务” KPI 加 1。 |

## 5. 测试环境与测试数据准备

- **环境要求**：使用 Docker Compose 启动的隔离测试环境（PostgreSQL, Redis, 前后端服务），不可连接生产数据库。
- **数据准备 (Seed Data)**：
  - 测试账号：3组 (Admin, Engineer, Viewer)。
  - 产品线：2个 (DC-DC-100, MOTOR-01) 用于交叉测试。
  - 标准算法数据集：预先录入 5 组 Minitab 提供的 SPC 示例数据和 MSA GR&R 原始数据，用于算法引擎测试断言。

## 6. 测试通过与验收标准 (Exit Criteria)

1. 所有 P0/P1 优先级测试用例 (Core Business Workflows) **100% 执行且通过**。
2. MSA 与 SPC 的所有数学计算、判异引擎计算结果与第三方标杆软件 (Minitab 等) 对比，误差率 `< 0.01%`。
3. 未修复的缺陷 (Bug) 中，不允许存在 `Critical` 与 `High` 级别的业务流程阻塞。
4. 全量验证产品线(Product Line) 数据权限，未发生跨产品线的数据泄露。
5. AuditLog 对重要审批与实体修改的覆盖率达到 100%。
