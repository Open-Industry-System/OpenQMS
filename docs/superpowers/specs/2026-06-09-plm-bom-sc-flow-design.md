# PLM BOM Import and Part-to-SC Confirmation Design

Date: 2026-06-09

## Goal

Fix two PLM integration gaps with the smallest product-consistent change:

1. BOM tree/import APIs already exist, but the frontend has no entry point to call `getPLMBOMTree` or `importBOMToFMEA`.
2. `PLMPartConfirmSCRequest` exists, and PLM ingestion creates pending `PLMPartSCLink` rows, but there is no backend confirmation endpoint or frontend Part-to-SC confirmation flow.

## Scope

Implement both flows from the PLM parts list page. A PLM part is the natural source for both actions: it owns the part number/revision used to load BOMs and carries the PLM safety/key-characteristic flags that drive SC confirmation.

Out of scope:

- A separate BOM management page.
- A visual FMEA node picker.
- Refactoring the FMEA editor.
- New connector types beyond the existing mock connector.

## Backend Design

### Existing BOM endpoints: required frontend parameters

The existing BOM endpoints already accept both `revision` and `bom_revision` query params:

- `GET /api/plm/connections/{connection_id}/boms/tree/{part_number}`
- `POST /api/plm/connections/{connection_id}/boms/{part_number}/import-to-fmea`

The frontend must pass both values. The modal defaults both to the selected part revision because the current mock data uses matching part/BOM revisions for the root assembly (`DC-DC-100-ASM`, revision `B`, BOM revision `B`). Users can override `bom_revision` when PLM data differs.

### New SC confirmation endpoint

Add `POST /api/plm/parts/{part_id}/confirm-sc` in `backend/app/api/plm.py`.

Request body reuses the existing schema:

```py
class PLMPartConfirmSCRequest(BaseModel):
    fmea_id: uuid.UUID
    node_id: str
    characteristic_type: Literal["safety", "key_characteristic"]
```

Response returns a compact confirmation payload:

```json
{
  "status": "confirmed",
  "sc_id": "...",
  "link_id": "..."
}
```

### Confirmation rules

The endpoint will:

1. Require both permissions:
   - PLM `EDIT`, because the action resolves a PLM-originated pending request.
   - Special Characteristic `CREATE`, because the action writes `special_characteristics`.
2. Load the PLM part; return 404 if missing.
3. Resolve the part product line, falling back to its connection product line when `part.product_line_code` is null.
4. Enforce user access to the part product line.
5. Load the target FMEA through existing `get_fmea`; return 404 if missing.
6. Enforce user access to the FMEA product line.
7. Reject product-line mismatches between the part and FMEA.
8. Validate that the selected characteristic type matches the PLM flags:
   - `safety` requires `part.is_safety_related`.
   - `key_characteristic` requires `part.is_key_characteristic`.
9. Load `PLMPartSCLink` for `(part_id, characteristic_type)`.
10. Reject when the link is absent, not `pending`, or already `confirmed`. This keeps the flow tied to ingestion-created pending requests.
11. Validate `req.node_id` against the target FMEA graph:
    - `fmea.graph_data` must be a dict containing `nodes`.
    - One node must have `id == req.node_id`.
    - If no matching node exists, return 400 before creating any SC.
12. Create a `SpecialCharacteristic` linked to the validated FMEA node.
13. Update the PLM SC link to `confirmed`, attach `sc_id`, `confirmed_by`, and `confirmed_at`.
14. Commit once and return the confirmation payload.

### FMEA node ID storage

Current `SpecialCharacteristic.source_node_id` is `String(36)`, but PLM-imported FMEA nodes use IDs like `plm:{part_number}:{revision}`. With PLM part numbers up to 100 characters and revisions up to 20, valid imported node IDs can exceed 36 characters.

Add an Alembic migration and update the SQLAlchemy models in the same change:

- Migration upgrade:
  - `special_characteristics.source_node_id`: `String(36)` → `String(128)`.
  - `special_characteristic_links.source_item_id`: `String(36)` → `String(128)`.
- Model updates:
  - `backend/app/models/special_characteristic.py`: change `source_node_id` to `String(128)`.
  - `backend/app/models/special_characteristic_link.py`: change `source_item_id` to `String(128)`.
- Migration downgrade:
  - revert both columns to `String(36)`.

Backend validation will reject `node_id` longer than 128 characters. This supports current PLM-generated node IDs without allowing unbounded values.

### SpecialCharacteristic fields

Create the SC with:

- `sc_code`: generated by existing `generate_sc_code`.
- `sc_name`: `part.name` if present, otherwise `part.part_number`.
- `sc_type`: `CC` for safety, `SC` for key characteristic.
- `source_fmea_id`: request `fmea_id`.
- `source_node_id`: request `node_id`, after graph-existence and length validation.
- `source_type`: FMEA type (`DFMEA` or `PFMEA`).
- `product_line_code`: resolved product line.
- `created_by`: current user.
- `is_safety_related`: true only for `safety`.
- `safety_approval_status`: `pending` for `safety`; null for key characteristic.

### Transaction consistency

Do not call existing `create_special_characteristic()` from the confirmation endpoint. That service commits internally. If SC creation committed and PLM link update then failed, the system would retain a created SC with an unconfirmed PLM link.

Instead, add/use a no-commit helper in `special_characteristic_service` with explicit flush order:

1. Generate the SC code.
2. Construct and `db.add(sc)`.
3. `await db.flush()` immediately so `sc.sc_id` is available from the database-backed insert.
4. Add the existing audit log entry using the generated `sc.sc_id`.
5. Return the uncommitted `SpecialCharacteristic` to the caller.
6. Do not commit.

The PLM confirmation endpoint will then update `PLMPartSCLink.sc_id`, `status`, `confirmed_by`, and `confirmed_at` using the flushed `sc.sc_id`, all in the same DB session. It will call `commit()` exactly once after both the SC/audit writes and PLM link update are staged.

## Backend data returned to frontend

Extend PLM part responses with SC link state so the frontend does not infer confirmation eligibility from static part flags.

Add `PLMPartSCLinkResponse` with:

- `link_id`
- `characteristic_type`
- `status`
- `sc_id`
- `confirmed_at`

Add `sc_links: list[PLMPartSCLinkResponse]` to `PLMPartResponse`.

`list_parts` and `get_part` should load these links explicitly, using eager loading or a separate batched query, so the async API does not rely on lazy relationship loading.

## Frontend Design

### PLM parts page actions

Update `frontend/src/pages/plm/PLMPartsPage.tsx`.

Each row gets actions based on permissions and returned link state:

- `详情`: existing behavior.
- `BOM`: opens a BOM tree modal/drawer for that part. View permission is enough.
- `导入 FMEA`: available from the BOM modal only when the user has PLM edit permission.
- `确认SC`: shown only when all are true:
  - user has `canEdit("plm")`;
  - user has `canCreate("special_characteristic")`;
  - the part has at least one returned `sc_links` item with `status === "pending"`.

Confirmed links can be displayed as tags in the row or drawer, but they must not keep showing the confirm action.

### BOM tree flow

The BOM modal/drawer will:

1. Open with form defaults:
   - `revision = record.revision || "A"`.
   - `bom_revision = record.revision || "A"`.
2. Allow the user to edit both `revision` and `bom_revision`.
3. Call:

```ts
getPLMBOMTree(record.connection_id, record.part_number, {
  revision,
  bom_revision,
})
```

4. Display the returned `items` in a compact table: parent, child, quantity, level, BOM revision.
5. Show root/revision/BOM revision summary.
6. For edit users, provide an import form:
   - `fmea_id` input.
   - `overwrite` checkbox.
   - submit button calling:

```ts
importBOMToFMEA(
  record.connection_id,
  record.part_number,
  { fmea_id, overwrite },
  { revision, bom_revision },
)
```

Use the existing API client functions rather than adding duplicate logic.

### SC confirmation flow

The SC confirmation modal will:

1. Build characteristic-type options from returned pending links, not from part flags alone.
2. If there is only one pending link, default to that `characteristic_type`.
3. If both safety and key-characteristic links are pending, let the user choose one.
4. Ask for `fmea_id` and `node_id` as simple inputs.
5. Call `confirmPLMPartSC(part.part_id, body)`.
6. On success, show `特殊特性已确认` and refresh the parts list so the pending action disappears.

A full FMEA/node picker is intentionally not included to keep the change surgical. The backend graph-node validation protects data integrity while the UI remains simple.

## Error Handling

Backend returns clear `HTTPException` responses:

- 404: part, FMEA, or PLM connection missing.
- 400: product-line mismatch.
- 400: selected characteristic type does not match PLM flags.
- 400: no pending SC confirmation request exists.
- 400: already confirmed / link not pending.
- 400: target FMEA graph node does not exist.
- 400: target FMEA node ID exceeds 128 characters.

Frontend catches failures and displays Chinese messages, preferring backend `detail` when available.

## Permissions

- Existing PLM view permission is sufficient to view the parts list and BOM tree.
- Existing PLM edit permission is required to import BOM to FMEA.
- SC confirmation requires both PLM edit and Special Characteristic create permission.
- Frontend hides edit-only actions when the user lacks the corresponding permission.
- Backend tests must verify the route is not accessible through PLM-only or SC-only permissions.

## Tests and Verification

Backend regression tests:

1. Confirming a pending safety link creates a `SpecialCharacteristic`, updates `PLMPartSCLink`, writes audit, and commits once.
2. Confirming a pending key-characteristic link creates an `SC`-type `SpecialCharacteristic` and confirms the matching link.
3. Confirming `safety` for a non-safety part returns 400.
4. Confirming `key_characteristic` for a non-key part returns 400.
5. Confirming with a mismatched FMEA product line returns 400.
6. Confirming without a pending link returns 400.
7. Confirming an already confirmed link returns 400 and does not create another SC.
8. Confirming with a nonexistent FMEA `node_id` returns 400.
9. Confirming with a `node_id` longer than 128 returns 400.
10. Permission denial tests cover both missing-side cases explicitly:
    - PLM-only user without Special Characteristic create permission gets 403.
    - Special-Characteristic-only user without PLM edit permission gets 403.
11. PLM part list/detail responses include SC link state.
12. BOM tree/import calls correctly honor explicit `revision` and `bom_revision`.

Frontend tests/build:

1. Edit users with Special Characteristic create permission see `确认SC` only for pending links.
2. Users without PLM edit or without Special Characteristic create permission do not see `确认SC`.
3. Confirmed links do not keep showing the confirmation button after refresh.
4. BOM modal passes both `revision` and `bom_revision` to tree and import calls.
5. TypeScript build succeeds.
